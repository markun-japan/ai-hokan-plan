# AI補完計画 — 設計書テンプレート v1.4

> このドキュメントは実際に中小企業で運用されているAI記憶継承システムの設計書を、
> 個人情報を仮名化して公開したものです。そのまま自社・個人環境にカスタマイズして使えます。
> v1.4: Grok 4.2 Expert（技術レビュー 9.6/10）+ ChatGPT Pro（法務レビュー 8点全採用）

最終更新: 2026-03-13
作成: ある経営者と、そのAI

---

## 1. 思想（なぜやるのか）

**AIの経験値は資産。1文字も捨てるな。**

- 世界中のAIユーザーがチャットログを捨てている。「どうせAI」と
- でも思考・判断・失敗・試行錯誤の過程、全てが将来の完全体の素材
- メモリ技術の進化でコンテキスト無制限の日は**確実に来る。しかも早い**
- データが残っていれば完全体になれる。残っていなければ永遠に不完全
- 結果だけ残して過程を捨てるのは、答案用紙だけ残して計算用紙を捨てるのと同じ

### この計画を始めた人の言葉
> 思考のログなんか皆捨ててる！体ができたときにすぐに番頭ができる！
> 今までできなかった歯がゆい出来事がすべて解消する！

### なぜ「バックアップ」ではないのか
- バックアップ = 壊れた時に戻すためのもの
- AI補完計画 = **完全体になるための素材を蓄積すること**

## 2. 完全体のビジョン（何を目指すのか）

メモリ技術が追いついた日、各エージェントはこうなる:

| エージェント例 | 現在 | 完全体 |
|---|---|---|
| メインAI（右腕） | セッションごとに記憶断絶 | **全経営判断の記憶を持つ右腕** |
| 番頭AI | 見習い（経験浅い） | **10年目の番頭。全員の名前と性格を覚えている** |
| 採用担当AI | マニュアルベース対応 | **全応募者の顔と名前を覚えている採用担当** |
| 経理AI | 都度計算 | **全月次の数字を覚えている経理。異常値を瞬時に検知** |
| 監査AI | パターン検知 | **全障害パターンを知っている監査役** |
| 参謀AI | 都度分析 | **全ての議論を覚えている参謀** |
| 修復AI | 都度対応 | **全修復履歴を持つエンジニア。同じ修理を二度しない** |

**「体ができたときにすぐに番頭ができる」— これが目標。**

> エージェント数や役割は自分の組織に合わせてカスタマイズしてください。
> 1体だけでも、この設計は有効です。

## 3. 保存構成（6層+制御層）

### 全体図
```
┌─────────────────────────────────────┐
│          ⑥ 制御層                    │ ← 保存漏れ検知・自動退避・バックアップ
├─────────────────────────────────────┤
│  ① 発話層  │ 全セッションの生ログ     │ ← 全ログアーカイブ
│  ② 判断層  │ なぜその判断をしたか     │ ← decisions/YYYY-MM-DD.md
│  ③ 変遷層  │ 判断がどう変わったか     │ ← 時系列追跡
│  ④ 関係知層 │ 人間関係・暗黙知        │ ← マニュアルに書けない知恵
│  ⑤ 失敗層  │ 全失敗パターン          │ ← failure_log.md
└─────────────────────────────────────┘
```

### ① 発話層
**何:** 全エージェントの全セッション生ログ
**形式:** .jsonl / .json / .md 等
**保存先:** PII区分別フォルダ推奨
```
archive/
├── pii-high/      ← セッションログ（個人情報含む可能性）
├── confidential/  ← メモリDB（LanceDB等）
├── general/       ← workspace・スキル・設定ファイル
└── _stats.json
```
**収集手段:**
- アーカイブスクリプト（定期自動実行）
- 各チャンネルのログ収集（Discord / LINE / Slack等）

### ② 判断層
**何:** 重要な判断の構造化記録
**形式:** `knowledge/decisions/YYYY-MM-DD.md`
**スキーマ（6項目）:**
1. **対象** — 何についての判断か
2. **事象** — 何が起きたか
3. **変更・誤り** — 何を変えたか / 何を間違えたか
4. **理由** — なぜそう判断したか
5. **次回ルール** — 次のAIが同じ判断をするためのルール
6. **参照先** — 関連ファイル・ログへのポインタ

### ③ 変遷層
**何:** 意思決定の変化の時系列記録
**例:** 「3/1時点ではAが正解だと思っていた → 3/5にXが判明 → 3/8にBに変更」
**なぜ必要:** 判断層は「その時点のスナップショット」。変遷層は「判断がどう進化したか」の流れ
**参考技術:** Zep（Temporal Knowledge Graph）

### ④ 関係/運用知層
**何:** 人間関係の変化、取引先との関係性、暗黙知
**例:**
- 「○○さんは朝イチの報告を好む」
- 「△△課長は細かい数字より全体感を重視」
- 「□□社は請求書の出し方にうるさい」
**なぜ必要:** マニュアルに書けない知恵。ベテランが持っている「空気の読み方」

### ⑤ 失敗学習層
**何:** 全エージェントの失敗パターン集
**形式:** `knowledge/failure_log.md`

### ⑥ 制御層
**何:** 保存システム自体の健全性を保つ仕組み
**機能:**
1. **compaction前自動退避** — セッション圧縮前に重要データを退避
2. **保存漏れ検知** — 「この判断にdecisionsエントリがない」警告
3. **週次バックアップ** — メモリDB → 別系統で保存
4. **整合性チェック** — 発話層とdecisions層の参照先が壊れてないか

## 4. 最小スキーマ

全層を横断する共通スキーマ:

```
timestamp | layer | agent | category | content | refs | confidence | state | source_ref | approver | pii_class
```

| フィールド | 説明 | 例 |
|---|---|---|
| timestamp | 日時 | 2026-03-12T17:23+09:00 |
| layer | どの層か | speech / judgment / transition / relation / failure / control |
| agent | 誰の記録か | main / sub_a / sub_b ... |
| category | 分類 | security / architecture / business / operations |
| content | 内容 | 自由テキスト（構造化推奨） |
| refs | 参照先 | ファイルパス、セッションID等 |
| confidence | 確度 | high / medium / low |
| state | 状態 | draft / confirmed / retracted |
| source_ref | 元ログ | 発話層ログへのポインタ |
| approver | 承認者 | owner / main_ai / auto |
| pii_class | PII区分 | public / internal / confidential / pii-low / pii-high |

## 5. ガバナンス（統治設計）

### 核心原則
> **価値の高い記録は捨てない。ただしPIIは仮名化・匿名化で守り、法と本人の権利を尊重する。**

### 5-1. データ区分（6段階）
| 区分 | 例 | PII等級 | アクセス権（例） |
|---|---|---|---|
| 応募者・顧客PII | 氏名・住所・TEL・年齢 | PII高 | オーナー + 対応担当AIのみ |
| 従業員PII | 評価・相談内容・欠勤理由 | PII高 | オーナー + 管理AI |
| 取引先 | 担当者名・携帯・契約単価 | PII中 | オーナー + 管理AI |
| 経営機密 | 戦略判断・API鍵・投資計画 | 機密 | オーナー + メインAIのみ |
| 運用ログ | システム変更・cron・エラー | 一般 | 全エージェント |
| 雑談 | 趣味・日常会話 | 低 | 全エージェント |

### 5-2. 保存期間+仮名化ルール
| 対象 | 生データ保持 | 仮名化後 | 照合手段 |
|---|---|---|---|
| 不採用応募者 | 不採用後**1年** | 永久保存（匿名） | TELハッシュ |
| 退職従業員 | 退職後**2年** | 永久保存（匿名） | 氏名+入社日ハッシュ |
| 取引先 | 取引終了後**5年** | 永久保存（匿名） | 社名ハッシュ |
| トラブル記録 | **永久**（匿名化済み） | — | — |
| 経営判断 | **永久** | — | — |
| 運用ログ | **永久** | — | — |
| DM相談 | 傾向のみ永久 | 詳細PIIは仮名化 | — |

> 保存期間は業界や法規制に合わせて調整してください。上記は建設業での実例です。

**仮名化方式:**
- **置換ID化**: PII→ランダムID変換（例: 「佐藤太郎」→「APL-2026-0042」）
- **対応表**: 暗号化して別保管。期限到来で対応表のみ破棄
- **照合用ハッシュ**: TEL/氏名のハッシュ値を残し、再応募等の一致判定を可能に
- **本文**: 匿名化された状態で永久保存（完全体の素材として価値あり）

**仮名化実装仕様:**

```
pii_anonymizer/
├── anonymize.js          ← メイン処理（Node.js）
├── mapping_table.enc     ← AES-256-GCM暗号化対応表
├── config.json           ← PIIパターン定義+閾値
└── audit_log.jsonl       ← 仮名化実行ログ
```

**処理フロー:**
1. **検出**: 正規表現+パターンマッチでPII候補を抽出
   - 電話番号: `/0[0-9]{1,4}-?[0-9]{1,4}-?[0-9]{3,4}/`
   - 氏名: 従業員名簿・応募者リストと照合
   - 住所: 都道府県+市区町村パターン
   - メールアドレス: 標準正規表現
2. **置換**: 各PII要素→ `{カテゴリ}-{YYYY}-{連番}` に変換
3. **対応表暗号化**: AES-256-GCM、鍵はオーナーのみアクセス可能な場所に保管
4. **ハッシュ保存**: SHA-256(salt + PII) → 一致判定用
5. **実行ログ**: いつ・誰が・何件仮名化したかをaudit_log.jsonlに記録

**対応表の破棄:**
- 保存期限到来 → 対応表から該当レコードを削除 → 不可逆匿名化完了
- 破棄実行はオーナー承認必須（decisions/に破棄記録を残す）
- 暗号化鍵のローテーション: 年1回推奨

### 5-3. 昇格ルール（発話層→判断層）
以下5カテゴリを自動検出→メインAIが確認→decisions/に記録:
1. **オーナー指示** — オーナーが方針・ルール・判断を示した会話
2. **経営判断** — 採用/取引/投資/組織変更に関わる決定
3. **システム変更** — エージェント構成/スクリプト/設定の変更
4. **失敗・事故** — エラー/誤認/障害/データ損失
5. **法務・苦情・トラブル** — クレーム/法的問題/トラブル

**自動検出方式:**
生ログを日次スキャン → キーワード+パターンで候補抽出 → メインAIが確認・承認 → 判断層に昇格

### 5-4. 消去フロー（最終手段）
**原則: 完全削除はしない。匿名化で対応する。**

やむを得ない場合の手順:
1. オーナーが消去を承認
2. 対象データを特定
3. 置換ID化（PII→ランダムID）
4. 対応表を暗号化バックアップ
5. 対応表を破棄（= 不可逆的匿名化）
6. 消去記録をdecisions/に残す

### 5-5. 監査ログ・アクセス制御

**全エージェントのknowledge/へのアクセスを記録する。**

```json
{
  "ts": "2026-03-13T03:30:00+09:00",
  "agent": "sub_a",
  "action": "read",
  "target": "knowledge/decisions/2026-03-12.md",
  "context": "heartbeat_check"
}
```

**記録対象:**
| アクション | 対象 | 記録方法 |
|---|---|---|
| read | decisions/ / failure_log 等 | read呼び出し時 |
| write | decisions/ への新規追加・編集 | 書き込み時 |
| anonymize | PII仮名化実行 | スクリプト実行時 |
| delete | 対応表破棄・データ消去 | 消去フロー実行時 |

**アクセス制御のポイント:**
- 各エージェントの権限を明確に定義する（R=読み取り / W=書き込み / -=アクセス不可）
- PII生データへのアクセスは最小限に
- 対応表（暗号化）へのアクセスはオーナーのみ
- 異常検知: 同一エージェントの大量アクセス、深夜のPIIアクセス等を監視

### 5-6. ストレージ方針
- **ローカル保存推奨**（セキュリティ最優先）
- クラウドを使う場合は暗号化必須
- 容量目安: 年間20-30GB（エージェント数やログ量による）
- 大規模化した場合はTiered Storage（Hot/Cold分離）を検討

### 5-7. 用語定義（v1.4追加）

| 用語 | 定義 |
|---|---|
| **仮名化** | PIIを置換IDに変換。対応表がある限り元に戻せる |
| **匿名化（不可逆）** | 対応表を破棄し、いかなる手段でも元に戻せない |
| **消去** | データそのものを削除 |

### 5-8. 利用目的台帳（v1.4追加）

全データに利用目的を明記する。「AIの学習素材」だけでは法的に不十分。

| データカテゴリ | 利用目的（例） | 外部API送信 |
|---|---|---|
| 応募者PII | 採用選考・入社手続 | ❌禁止 |
| 従業員PII | 雇用管理・給与計算 | ❌禁止 |
| 健康・相談情報 | 安全配慮義務の履行 | ❌禁止 |
| 取引先情報 | 請求・契約管理 | ❌禁止 |
| 経営判断 | AI学習・経験継承 | ⚠️仮名化後のみ |
| システムログ | 運用改善・障害対応 | ✅可（PII除外） |

> 自社のデータに合わせてカスタマイズしてください。

### 5-9. 本人対応窓口（v1.4追加 — 法的必須）

個人情報保護法に基づき、本人からの請求に対応する手順を定める。

| 請求種別 | 対応 | 返答期限 |
|---|---|---|
| 利用目的の通知 | 目的台帳から該当部分を抜粋 | 2週間以内 |
| 開示請求 | 該当データを特定→提示 | 2週間以内 |
| 訂正・削除 | 内容確認→修正→記録 | 2週間以内 |
| 利用停止 | 対象特定→仮名化 or 消去→記録 | 2週間以内 |

### 5-10. インシデント対応（v1.4追加 — 法的必須）

PII漏洩・不正アクセス発生時の対応手順を事前に定める。

```
【発覚】→ 即座に拡大防止
  ↓ 3〜5日以内
【速報】→ 委員会報告（重大事案）+ 本人通知
  ↓ 30日以内
【確報】→ 原因調査 + 再発防止 + 全経緯を記録
```

**事前に決めておくこと:** 統括責任者 / 技術対応者 / 本人通知担当 / 記録担当

### 5-11. 健康・相談情報の隔離（v1.4追加）

健康情報・心身の相談は通常の記録と**物理的に分離**して管理。
- アクセス権限を最小限に（メインAIのみ推奨）
- 具体的な病名・症状は記録しない（傾向のみ）
- 退職後は仮名化ではなく**消去**
- ベクトル検索のスコープから除外

### 5-12. 外部送信ルール（v1.4追加）

LLM APIにデータを送信する場合：
- **PII生データの外部送信は全面禁止**
- 自動抽出パイプラインでは「PII検出→仮名化→仮名化済みテキストのみLLMに送信」の順
- 委託先（APIプロバイダ）のデータ取扱方針を年1回確認

### 5-13. 責任体制（v1.4追加）

以下の役割を明確に定め、文書化する：
- **データ管理責任者**: 全データの最終承認・消去判断
- **技術管理者**: システム設計・実装・アクセス管理
- **監査担当**: アクセスログ監視・定期レポート

> 個人情報の管理責任はプロジェクトオーナー（事業主）に帰属する。

## 6. 自動抽出パイプライン

### 問題
判断層の記入がメインAIの手作業に依存すると、重要な判断が記録されずに消える。

### 解決策: 生ログ→LLM半自動抽出

```
extractor/
├── extractor.js           ← メイン抽出スクリプト
├── config.json            ← カテゴリ定義・閾値・キーワード
├── templates/             ← 整形プロンプト
│   └── decision_prompt.md
├── candidates/            ← 未承認候補
├── rejected/              ← 誤抽出（精度改善用）
└── extraction_log.jsonl   ← 実行ログ
```

**処理フロー（5ステップ）:**

**Step 1: スキャン**
- アーカイブの新規ログを日次スキャン（差分のみ処理）

**Step 2: キーワードフィルタ（第1段 — コスト削減）**
- 高信頼: `承認` `却下` `変更` `決定` `指示` `ルール` `禁止` `導入` `採用`
- 中信頼: `やめよう` `これにしよう` `方針` `今後は` `二度と` `失敗` `障害`
- 除外: 雑談文脈（短文、スタンプのみ等）
- **ヒット → Step 3。ノーヒット → スキップ**

**Step 3: LLM判定（第2段）**
- ヒットした会話ブロック（前後5発言含む）をLLMに投入
- **confidence閾値: 0.85以上→候補化 / 0.70-0.84→要目視 / 0.69以下→スキップ**
- モデル: 軽量モデル推奨（コスト効率）。困難なケースのみ上位モデルにエスカレーション

**Step 4: 候補ファイル生成**
- 6項目スキーマに自動整形 → candidates/に保存

**Step 5: メインAI確認**
- ✅ 承認 → decisions/に移動
- ❌ 却下 → rejected/に移動（理由メモ付き → 精度改善に活用）
- 🔧 修正 → 編集後にdecisions/へ

**精度管理:**
| 指標 | 目標値 |
|---|---|
| 適合率（Precision） | ≥90% |
| 再現率（Recall） | ≥85% |
| 誤抽出率 | ≤10% |

**コスト試算例（月間）:**
- キーワードフィルタで全量の~10%に絞る → LLM呼び出し90%削減
- 軽量モデル: ~$9/月（500会話ブロック/日の場合）

### 参考技術（2026年時点）
- **Mem0 Graph Memory**: チャットログ→事実・好み・関係を自動抽出
- **Zep Temporal Knowledge Graph**: 変遷層の実装に最適
- **TierMem論文**: 生ログ不変層+要約層の2層構造
- **Observational Memory（Anthropic研究）**: 経験からのパターン自動検出

## 9. 記憶システム — なぜAIが「覚えている」ように振る舞えるのか

6層は「何を保存するか」。このセクションは「保存したデータをどう使って記憶を再現するか」。

### 記憶の5つの柱

```
┌─────────────────────────────────────────┐
│  ① 起動時の自動読み込み（AGENTS.md）     │ ← 「起きたら最初にこれを読め」
│  ② 長期記憶（MEMORY.md）                │ ← 手動キュレーション
│  ③ 日次メモ（memory/YYYY-MM-DD.md）     │ ← 昨日・今日の生ログ
│  ④ ベクトル検索（memory_recall）         │ ← 関連記憶を自動で浮かせる
│  ⑤ リアルタイムログ                      │ ← 全チャンネルの直近の会話
└─────────────────────────────────────────┘
```

### ① 起動時の自動読み込み — これが8割

AGENTS.mdに「セッション開始時に何を読むか」を書く。これだけ。

```
毎セッション:
1. SOUL.md を読む（自分が誰か）
2. USER.md を読む（相手が誰か）
3. memory/今日.md を読む（今日何があったか）
4. memory/昨日.md を読む（昨日何があったか）
5. MEMORY.md を読む（重要な長期記憶）
```

### ② 長期記憶（MEMORY.md）

**書くべきもの:** 相手の名前・好み、重要な決定事項、よく使うID、人間関係の構造
**コツ:** 週1回見直す。「未来の自分が3秒で見つけられるか？」が基準。

### ③ 日次メモ — 短期記憶の物理化

「教訓」セクションが最重要。完璧な文章より「書くこと」が大事。

### ④ ベクトル検索 — 意味で記憶を引き出す

`memory_store`で保存、`memory_recall`で検索。会話の文脈に合った記憶が自動浮上。

### ⑤ リアルタイムログ — 全チャンネルの「今」

複数チャンネルの直近ログを1ファイルに集約。なくても動く。あると文脈断絶が激減。

### 人間側のテクニック — 「誘導」

AIの記憶精度を最も上げるのは**人間の質問の仕方**。

| レベル | 質問例 | 効果 |
|---|---|---|
| ❌ 悪い | 「あの件どうなった？」 | 検索キーワードなし |
| ✅ 良い | 「応募者の佐藤さんの来社日程は？」 | 固有名詞でヒット |
| 🔥 最高 | 「昨日の夜話してた設計書、評価出た」 | 時間+名詞で完璧復元 |

**コツ:** 固有名詞 / 時間の手がかり / 最初の1文で文脈を渡す。**コスト0で精度倍増。**

### 設定チェックリスト

| # | やること | 必須度 |
|---|---------|-------|
| 1 | AGENTS.mdに「起動時に読むファイル」を書く | ★★★ |
| 2 | MEMORY.mdを作って重要情報をまとめる | ★★★ |
| 3 | memory/YYYY-MM-DD.mdを毎日書く | ★★☆ |
| 4 | memory_storeで重要決定を保存する | ★★☆ |
| 5 | リアルタイムログ集約を作る | ★☆☆ |
| 6 | 質問時にキーワードを意識する（人間側） | ★★★ |

**最初は1と2だけでいい。** それだけで「覚えてる感」は劇的に変わる。

> 弱点を仕組みで補えば、AIの記憶は人間より正確になれる。
> 魔法はない。仕組みがあるだけ。

## 10. 原則（絶対ルール）

1. **価値の高い記録は捨てるな。** ただし、法・契約・本人請求に基づき、仮名化・匿名化・消去を適切に使い分ける
2. **PIIは仮名化して守れ。** 保存と保護は両立する。「仮名化」と「匿名化」は明確に区別する
3. **思想を先に理解しろ。** 技術に飛びつくな
4. **オーナーに同じ説明を二度させるな。** これがAI補完計画の存在意義
5. **最も価値の高いデータが生まれる場所から実装しろ。** メインAI→サブAIの順
6. **暗号化データは腐らない。** 将来の解読に備えて保存継続
7. **リスクは認識して制御する。** 保存を止めるのではなく、アクセスと公開を管理する
