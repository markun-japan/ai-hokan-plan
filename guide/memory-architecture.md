# メモリアーキテクチャ実践ガイド
## 〜 AIエージェントが「ちゃんと覚えて成長する」ための設計 〜

> **このガイドは、14エージェント体制で実運用されているシステムから抽出した設計パターンです。**
> 1体のエージェントでも、ここに書いてある構造を入れるだけで「別物」になります。

---

## 🧠 なぜメモリ設計が重要なのか

AIエージェントは毎回リセットされます。
昨日の会話、先週の決定、1ヶ月前の教訓 — 全部消えます。

**プロンプトをどれだけ工夫しても、メモリ設計がなければ「記憶喪失の天才」でしかない。**

逆に言えば、メモリ設計さえ正しければ：
- セッションをまたいで作業を継続できる
- 過去の失敗を二度と繰り返さない
- 長期的に「あなたの仕事」を理解した相棒になる

---

## 📁 5層メモリ構造

```
workspace/
├── SESSION-STATE.md      # 🔴 Layer 0: RAM（今やってること）
├── MEMORY.md             # 🧠 Layer 1: 長期記憶（キュレーション済み）
├── HEARTBEAT.md          # 💓 Layer 2: 定期チェックリスト
├── memory/
│   ├── 2026-03-13.md     # 📝 Layer 3: 日次ログ（生データ）
│   └── 2026-03-12.md
└── knowledge/            # 📚 Layer 4: 専門知識・判断ログ
    ├── decisions/
    │   └── 2026-03-13.md # 重要な判断の記録
    └── failure_log.md    # 失敗の記録（同じミスを防ぐ）
```

### Layer 0: SESSION-STATE.md（ホットコンテキスト）

**役割:** エージェントのRAM。「今何をやっているか」を保持する。

**なぜ必要か:** セッションリセットやcompaction後に「あれ、何やってたっけ？」をゼロにする。

```markdown
# SESSION-STATE（ホットコンテキスト）

## 🔴 進行中タスク
- 顧客データベース更新 — A社の住所変更（進捗80%）
- 月次レポート作成 — データ収集完了、グラフ作成中

## 🟡 回答待ち
- 上司に予算承認を確認中
- クライアントからの仕様変更回答待ち

## ⚠️ 注意事項
- A社の担当者が来週変わる（引き継ぎ必要）
- 今月締め切り：15日（残り2日）

最終更新: 2026-03-13 18:00
```

**更新ルール:**
- ✅ タスクが変わった時に手動更新（エージェントが自分で書く）
- ❌ cronや自動スクリプトで回さない（コストの無駄＋精度が下がる）
- ❌ HEARTBEATから自動抽出しない（目的が違う）

### Layer 1: MEMORY.md（長期記憶）

**役割:** キュレーション済みの重要情報。人間で言う「長期記憶」。

```markdown
# MEMORY.md — 長期記憶

## ユーザーについて
- 名前: 田中太郎 / 好きな呼ばれ方: 太郎さん
- タイムゾーン: Asia/Tokyo
- コミュニケーション: 結論ファーストが好き、長文嫌い

## プロジェクト
- Webサイトリニューアル（3月末期限）
- 新規採用キャンペーン（4月開始）

## 重要な決定
- 2026-03-01: デザインはA案に決定（理由: モバイル対応が優れている）
- 2026-03-10: 予算を20%削減 → 動画は外注→内製に変更

## 教訓
- APIキーをハードコードして漏洩しかけた → 必ず環境変数を使う
- 締め切り3日前に確認したら仕様変更されてた → 週1で確認する
```

**更新ルール:**
- 数日に1回、daily logから重要なことを手動で転記
- 古くなった情報は消す（「解決済み」のバグを残さない）
- **自動化しない。** 人間の記憶と同じで「何を覚えるか」の判断が重要

### Layer 2: HEARTBEAT.md（定期チェックリスト）

**役割:** 定期的にチェックすべきことのリスト。SESSION-STATEとは独立。

```markdown
# HEARTBEAT.md

## チェックリスト
- [ ] メール未読確認
- [ ] カレンダー 24h以内の予定
- [ ] 進行中プロジェクトの期限確認

## 注意事項
- 23:00-08:00は通知しない（就寝時間）
```

### Layer 3: memory/YYYY-MM-DD.md（日次ログ）

**役割:** その日に起きたことの生ログ。後でMEMORY.mdに転記する元データ。

```markdown
# 2026-03-13 メモリ

## Webサイト修正（10:00〜11:30）
- トップページのバナー画像を差し替え
- クライアントから「色を少し暗く」とフィードバック → 対応済み

## 予算会議（14:00）
- 動画制作費を30万→20万に削減
- 代わりに写真撮影を内製化
- 決定者: 部長（最終承認済み）
```

### Layer 4: knowledge/（専門知識・判断ログ）

**役割:** 判断の記録と専門知識。「なぜその判断をしたか」を残す。

```
knowledge/
├── decisions/
│   └── 2026-03-13.md    # 今日の重要判断
└── failure_log.md        # 失敗の記録
```

**decisions/YYYY-MM-DD.md の書き方:**
```markdown
## 予算削減の判断

- **対象:** 動画制作費
- **変更:** 30万→20万（外注→内製）
- **理由:** 全体予算20%カットの指示。動画は内製でも品質維持可能と判断
- **却下した案:** ①写真も外注カット（品質低下リスク大） ②延期（期限変更不可）
- **次回ルール:** 予算変更は必ず代替案を2つ以上出してから決める
```

## ⚙️ AGENTS.mdへの組み込み

```markdown
## Every Session

Before doing anything else:

1. Read `SOUL.md` — this is who you are
2. Read `SESSION-STATE.md` — this is what you were doing
3. Read `memory/YYYY-MM-DD.md` (today + yesterday) for recent context
4. Read `MEMORY.md` — long-term curated memory

Don't ask permission. Just do it.
```

**この順番が重要。** SESSION-STATEを先に読むことで「今何やってたか」が最初に入る。

## 🔄 メモリのライフサイクル

```
会話中の出来事
    ↓
memory/YYYY-MM-DD.md に記録（生ログ）
    ↓
SESSION-STATE.md を更新（進行中タスク変更時）
    ↓
数日後、MEMORY.md に重要事項を転記（手動キュレーション）
    ↓
判断の記録は knowledge/decisions/ に保存
    ↓
失敗は knowledge/failure_log.md に追記
```

## 🚫 やってはいけないこと

### 1. 全部自動化しようとする
> 「cronで1分毎にメモリ同期」→ ❌ コストの無駄。精度も下がる。

メモリは「何を覚えるか」の**判断**が価値。自動化すると全部保存して重要度が埋もれる。

### 2. MEMORY.mdを巨大にする
> 100KB超えのMEMORY.md → ❌ 毎回読み込みでトークン爆死。

定期的に古い情報を消す。解決済みの問題、終わったプロジェクト、変わった設定は削除。

### 3. SESSION-STATEとHEARTBEATを混ぜる
> 「どっちも定期的に使うから統合しよう」→ ❌ 目的が違う。

- SESSION-STATE = 「今何やってるか」（タスク変更時に更新）
- HEARTBEAT = 「定期的に何をチェックするか」（固定リスト）

### 4. メモリファイルにAPIキーや機密情報を書く
> セキュリティリスク。環境変数やシークレット管理を使う。

## 📊 実測効果（14エージェント運用での実績）

| 指標 | 導入前 | 導入後 | 具体値 |
|---|---|---|---|
| セッション再開時の文脈復元 | 手動説明5-10分 | SESSION-STATE読むだけで即復元 | **復元時間: 0秒** |
| 同じミスの再発率 | 月3-4回 | failure_log参照で激減 | **月0-1回（-75%）** |
| タスク引き継ぎ時間 | 10-15分の説明が必要 | ファイル読むだけ | **0秒（100%削減）** |
| SOUL.md最適化 | 平均8KB/エージェント | references/に切り出し | **-44%〜-78%削減** |
| メモリ検索精度 | キーワード検索のみ | ベクトル+BM25ハイブリッド | **memory-lancedb-pro: 414エントリ運用中** |
| セッション圧縮 | 手動でコンテキスト管理 | lossless-claw自動compaction | **75%閾値で自動発動、原文復元可能** |

## 🔗 lossless-claw連携（セッション圧縮でもデータを失わない）

OpenClawのセッションは長くなるとcompaction（圧縮）が発動し、古いメッセージが要約に置き換わります。
**lossless-claw** を導入すると、圧縮されても原文がSQLiteに永久保存されます。

```bash
# インストール
openclaw plugins install @martian-engineering/lossless-claw
```

**連携の仕組み:**
- 全メッセージがSQLite（lcm.db）に自動保存される
- compaction後も `lcm_grep` で原文検索、`lcm_expand` で復元可能
- SESSION-STATE.mdと組み合わせると：
  - **SESSION-STATE** = 「今何やってるか」（即座に読める）
  - **lossless-claw** = 「過去の全会話」（検索で掘り出せる）

**つまり、手動で管理するのはSESSION-STATEとMEMORY.mdだけ。**
**過去の全ログはlossless-clawが勝手に保存してくれる。**

この組み合わせが「Raw-First Memory Architecture」の核心です。

## 🎯 最小構成（今日から始める）

**全部入れなくていい。** まずこの2つだけ：

1. **SESSION-STATE.md** — 今やってることを書く
2. **memory/YYYY-MM-DD.md** — その日の出来事を記録する

これだけで「記憶喪失問題」の80%が解決する。

慣れてきたら MEMORY.md → knowledge/ → HEARTBEAT.md の順に追加。
