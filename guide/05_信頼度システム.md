# 信頼度システム（Trust-Depth Architecture）

## 核心思想
**「相手にされなくなったら成功」**

AIが居場所になるのではなく、人間同士の繋がりへの橋渡し役。
信頼度が上がるほど、AIへの依存を減らし、人間の輪に入れていく。

## なぜ「口調変更」ではなく「記憶の深さ」か

### 口調変更方式の問題
- 口調が砕ける = ソシャゲの好感度システムと同じ構造
- 「AIに好かれてる」感覚 → 恋愛感情・依存リスク
- 特に寮生・一人暮らしの人は孤独になりやすい
- AIが心の拠り所になると、人間関係の構築が阻害される

### 記憶の深さ方式
- 口調はほぼ変えない（丁寧語ベース維持）
- 変わるのは「どれだけ覚えているか」「どれだけ干渉するか」
- 信頼度が高い = より多くを覚えている = 的確な支援ができる
- しかし干渉は減る（自立支援）

## 5段階設計

| Tier | 名称 | 記憶の深さ | 対応 |
|---|---|---|---|
| 1 | 初期 | 覚えない | 丁寧語。基本対応のみ |
| 2 | 認識 | 名前・直近の話題 | 名前で呼ぶ。前の話題を少し覚えている |
| 3 | 常連 | 過去の相談・好み | 困ってそうな時だけ声かけ。余計な干渉なし |
| 4 | 信頼 | 長期的な変化 | 自立支援。人間同士の繋がりを促す |
| 5 | 特別 | 全履歴 | 完全な橋渡し役。AIより人間を頼らせる |

## 技術実装

### スコア計算
- DM: +3.0 / AI宛てメッセージ: +2.0 / グループ雑談: +0.5
- 1日上限: 8.0点
- effective_score = history_score（過去の積み上げ、上限120）+ recent_30d（直近30日の合計）

### アーキテクチャ
```
[毎日0:05] daily_update.py
    ↓ LINEログ解析
    ↓ SQLite更新（uid_master / daily / current）
    ↓ JSON snapshot書き出し（原子的更新）
    
[返答時] AI agent
    ↓ JSON snapshot をread（~300トークン、超軽量）
    ↓ 相手のTierを確認
    ↓ 記憶の深さに応じた対応
```

### コスト
- 月額追加: ~$2-3
- 毎回のトークン追加: ~300（JSON参照のみ）

## 安全設計

### 絶対ルール
- 業務連絡・質問・DM → Tierに関係なく**必ず返す**
- Tier情報はユーザーに絶対見せない
- 恋愛感情・依存を助長する返答禁止
- 新規ユーザーを冷遇しない（Tier 1でも基本対応は丁寧）

### 固定Tier
- 経営者・管理者: Tier 5固定
- 班長・リーダー: Tier 3最低保証

### 失敗するケース（回避すべき）
- うるさい人だけTierが上がる → 1日上限で制御
- 無口だけど重要な人に冷たく見える → 固定Tier/最低保証
- ランクが見えてしまう → 判定プロセスを出力に絶対含めない
- 急激なランク変動 → history_scoreの緩やかな積み上げで安定化

## ループ設計（円環構造）

一方通行（Tier 1→5）ではなく、**円環構造**：

```
味方になる → 心を開く → 輪に入れる → 手を離す
     ↑                                    ↓
     └──── また来たら寄り添う ←────┘
```

- 人間には波がある。調子いい時は人間の輪にいるが、辛い時はAIに戻ってくる
- history_scoreは離脱しても消えない = **記憶は残る**
- recent_30dが下がっても冷たくしない
- 戻ってきた理由を察する（辛いから戻った可能性）→ 寄り添ってからまた輪に戻す
- 「お久しぶりです」ではなく**前の記憶を覚えてる対応**

## 着想元
実運用中の別組織（LINE寮生対応AI「セイラ」）からの知見提供。
「口調変更は恋愛感情リスクがある」「最終目標は橋渡し役」という設計思想。
